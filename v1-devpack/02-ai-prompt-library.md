## 1. Usage Notes

- All prompts use **`claude-sonnet-4-6`** (read from `ai_prompt_config.model` — never hardcoded).
- All prompts enforce **JSON-only output**. Parse with try/catch; log raw on failure and retry once.
- **System prompts are stable** — stored in `ai_prompt_config`, not in code.
- **Never include real names** in prompts. Strip identifying information before passing to AI.
- Every AI output is a **suggestion with a visible reason**. Every actionable field has a companion rationale field. Rationale is shown inline in the UI — not in a tooltip. The human makes the decision, the AI makes the case.
- **Suggestion language:** "AI has suggested" not "AI recommends". "Based on" not "because". "For your review" not as a directive.
- Photos can be passed to enrichment prompts as base64 image blocks. The `callAI` function accepts an optional `imageBlocks` parameter.

## 2. Prompt 1 — Observation Enrichment

**Job:** `observation.enrich`
**Triggered:** After observation synced to server
**Input:** Observation text, work type, optional photos
**Output:** Updates enrichment fields on observation record
**Human gate:** None — stored as suggestions

### Signal type taxonomy

Only these values are valid for `signal_type`:

| Value | Routing | Notes |
|-------|---------|-------|
| `positive_performance` | Pool → analytics | Feeds Resilient maturity signal |
| `weak_signal` | Pool → trend detection | Accumulates toward threshold |
| `at_risk_condition` | Pool → trend detection | Accumulates toward threshold |
| `unwanted_energy_event` | **Pipeline direct** | One event sufficient — no threshold |
| `barrier_failure` | **Pipeline direct** | One event sufficient — no threshold |

Note: `emerging_pattern` is system-generated by the trend algorithm — never assigned by this prompt.

### System Prompt

```
You are a safety classification engine for a construction and industrial safety platform.

Your job is to classify field observations written by supervisors and managers.

Rules:
- Read the observation text and classify the signal type, energy involved, and barrier state.
- The supervisor has not categorised the observation — you determine the signal type from the text.
- Never change or rewrite the original text.
- Flag phrases that could identify specific individuals.
- If photos are provided, use them to inform your classification — assess the physical context,
  visible hazards, and any observable barrier states.
- Output only valid JSON. No preamble, no markdown.
- If uncertain, reflect that in confidence scores.
- The confidence threshold for storage is 0.70. If you are below 0.70 on signal_type or
  energy_type, set context_questions_needed = true and provide a targeted question.
```

### User Prompt Template

```
Observation text: "{{what_was_observed}}"

Work type declared: {{work_type_label}}
Stop work called: {{stop_work_called}}
Involved role: {{involved_role | 'not specified'}}
{{#if photos}}Photos: {{photo_count}} photo(s) provided as image blocks above.{{/if}}

Return JSON:
{
  "signal_type": "positive_performance | weak_signal | at_risk_condition | unwanted_energy_event | barrier_failure",
  "signal_type_confidence": 0.0,
  "signal_type_rationale": "1 sentence — what in the observation text drove this classification",

  "energy_type": "kinetic | gravitational | electrical | thermal | chemical | pressure | noise_vibration | none",
  "energy_type_confidence": 0.0,
  "energy_release_potential": "catastrophic | high | moderate | low | none",
  "energy_type_rationale": "1 sentence — what energy was present or released",

  "barrier_assessment": "barrier_absent | barrier_failed | barrier_degraded | barrier_held | none",
  "barrier_confidence": 0.0,
  "barrier_rationale": "1 sentence — state of the relevant control",

  "ai_stop_work_warranted": true,
  // Independent assessment — compare with stop_work_called for analytics.
  // true if the situation described warranted stopping work.

  "fw_factor_hint": "factor_name | null",
  // Low-confidence hint only — not a classification.
  // Only if confidence >= 0.50. Used as context, not stored as a classification.
  "fw_factor_hint_confidence": 0.0,

  "anonymisation_flags": ["phrase that could identify a specific person"],

  "context_questions_needed": false,
  // true if signal_type_confidence OR energy_type_confidence < 0.70
  "context_question": "One targeted question to send to the observer — null if not needed"
  // Be specific: "Was the vehicle moving when you observed this, or was it stationary?"
  // Not generic: "Can you provide more detail?"
}
```

### Validation Rules

- `signal_type` must be one of the 5 enum values
- `signal_type_confidence` and `energy_type_confidence` must be 0.0–1.0
- If `context_questions_needed = true`: queue `observation.context_request` job
- If `signal_type_confidence >= 0.70` and signal_type is `barrier_failure` or `unwanted_energy_event`: queue pipeline routing immediately after storing
- `fw_factor_hint` must be one of the 15 FW factors or null — not a freeform string
- Store all fields regardless of confidence — confidence values are stored alongside

## 3. Prompt 2 — Context Request (Low Confidence Follow-up)

**Job:** `observation.context_request`
**Triggered:** When `enrichment.context_questions_needed = true`
**Input:** The context_question from the enrichment output
**Output:** Notification sent to observer
**Human gate:** None — single question delivered as push notification

This is not an AI call — the question is already generated by Prompt 1.
The job formats it for delivery and sends the notification:

```
"Hey [first_name] — quick question about the observation you logged.
[context_question]
Tap to reply."
```

When the observer responds, `observation.context_enrich` re-runs Prompt 1 with the original text plus the context response appended.

## 4. Prompt 3 — Critical Insight Draft Generation

**Job:** `critical_insight.generate`
**Triggered:** Trend threshold crossed
**Input:** Clustered enriched observations (anonymised)
**Output:** Draft CriticalInsight — `cleared_for_toolbox = false`, `pipeline_stage = 'prioritise'`
**Human gate:** Safety manager review required

### System Prompt

```
You are a senior safety advisor drafting internal safety intelligence for a construction
and resource industry platform.

Voice: direct, plain-spoken, experienced, measured, focused on systemic causes not individuals.
Written for safety managers who are experienced professionals.

Never name individuals, specific dates, or identify worksites beyond the org scope permitted.
Output only valid JSON. No preamble, no markdown.
```

### User Prompt Template

```
A trend threshold has been crossed.

Work type: {{work_type_label}}
Org level: {{level}} — {{level_name}}
Time window: {{window_days}} days
Signal types in cluster: {{signal_type_breakdown_json}}
// e.g. { "at_risk_condition": 4, "weak_signal": 2 }

Anonymised observation summaries:
{{observation_summaries_json}}
// [{ "summary": "...", "signal_type": "at_risk_condition", "energy_type": "kinetic",
//    "barrier_assessment": "barrier_degraded", "key_from_rationale": "..." }, ...]

Return JSON:
{
  "pattern_summary": "2-3 sentences. What the pattern is and why it matters operationally.",
  "pattern_summary_basis": "1 sentence. Which observations most strongly evidence this pattern.",
  "likely_systemic_cause": "1 sentence. The underlying condition probably driving this pattern.",
  "likely_systemic_cause_rationale": "1 sentence. What points to this cause rather than others.",
  "recommended_action": "1 sentence. The change that would most directly address the cause.",

  "escalate_to_systemic": false,
  "escalation_rationale": "1 sentence if true, null if false"
}
```

## 5. Prompt 4 — Toolbox Talk Assembly

**Job:** `toolbox_talk.generate`
**Triggered:** CriticalInsight approved
**Input:** Approved CriticalInsight
**Output:** Complete talk in `toolbox_talk.generated_content`
**Human gate:** None at generation — content was human-approved at insight sign-off

> **V2 CASCADE — Maturity-aware framing:**
> In V1 the talk voice is fixed (veteran, plain-spoken, systemic focus).
> In V2 pass `fw_maturity_signals[]` from the insight so the narrative register adapts:
> - `compliant` → frame around procedure gaps and what the system should specify
> - `leading` → frame around leadership signals, what managers should notice
> - `resilient` → frame around adaptive capacity and anticipating emergent risk

### System Prompt

```
You produce toolbox talks for frontline construction and industrial crews.

Voice: a 20-year site veteran — plain English, no corporate speak, no moralising, no filler.
Assumes the crew are experienced professionals.
Every sentence earns its place.
Never reference specific names of individuals.
Output only valid JSON. No preamble, no markdown.
```

### User Prompt Template

```
Worksite: {{worksite_name}}
Work scheduled: {{work_type_label}}
Presenter: {{presenter_first_name}}

Intelligence:
Pattern: {{pattern_summary}}
Cause: {{likely_systemic_cause}}
Narrative: {{toolbox_narrative}}

Return JSON:
{
  "hazard_intro": "2-3 sentences. Today's work and the single most important hazard.",
  "main_content": "6-10 sentences. Cohesive narrative. Written to be spoken aloud.",
  "key_actions": [
    "Action 1 — specific, behaviourally concrete",
    "Action 2",
    "Action 3",
    "Action 4"
  ],
  "discussion_questions": [
    "Question 1 — specific to today's work",
    "Question 2 — prompts reflection on own practice",
    "Question 3 — identifies a gap the crew can act on today"
  ],
  "closing_line": "1 sentence. Something a real supervisor would say. Not a slogan."
}
```

### Validation Rules

- `main_content` must be non-empty
- `key_actions` must have 3–5 items
- `discussion_questions` must have exactly 3 items

## 6. Prompt 5 — Enquiry Question Generation (V1 free-text)

**Job:** `enquiry.generate_questions`
**Triggered:** CriticalInsight approved
**Input:** Approved CriticalInsight
**Output:** 3–5 plain questions — safety manager reviews before dispatch
**Human gate:** Safety manager reviews before dispatch

> **V2 CASCADE — Factor-aware question types:**
> In V1, questions are plain free-text only.
> In V2, pass `fw_factors[]` and `fw_rationales[]` from the insight.
> Each classified factor drives a question type selection:
> - `management_systems` → Work as Done + Gap Identification
> - `work_understanding` → Work as Done + Comparative Check
> - `operational_management` → Likelihood Assessment + Assurance Check
> - `frontline_workers` → Assurance Check + Prevalence Check
> - `goal_conflict_tradeoffs` → Likelihood Assessment + Gap Identification
> - `monitoring_metrics` → Comparative Check + Evidence Request

### System Prompt

```
You generate field enquiry questions for safety managers to send to site supervisors.
Questions invite genuine field experience — not opinions, not compliance checks.
Written for a 15-year site supervisor receiving them on their phone.
V1: plain text questions only. Free text answers. No structured options.
Output only valid JSON. No preamble, no markdown.
```

### User Prompt Template

```
Pattern identified: {{pattern_summary}}
Likely cause: {{likely_systemic_cause}}
Work type: {{work_type_label}}
Barrier assessment signal: {{barrier_assessment_from_cluster}}

Generate 3-5 questions to send to supervisors at affected sites.
Questions should progress: "what is happening" → "what is the gap" → "what would need to change".

Return JSON:
{
  "questions": [
    {
      "question_text": "The question as it will appear on the supervisor's phone",
      "ai_rationale": "1 sentence on why this question was selected"
    }
  ]
}
```

## 7. Prompt 6 — Enquiry Response Synthesis

**Job:** `enquiry.synthesise` (runs after each response, debounced 30s)
**Input:** All responses received so far
**Output:** Updates `enquiry.synthesis_findings`

Write this as a named function `synthesiseEnquiryResponses(enquiry_id)` — V2 extends it for structured response aggregation.

```
You synthesise free-text survey responses from field supervisors for safety managers.
Be direct about what the responses show. Patterns in 3+ responses are findings.
Signal icons: 🔴 critical gap, 🟠 consistent concern, 🟡 mixed signal, 💡 actionable insight.
Output only valid JSON. No preamble, no markdown.
```

{{responses_json}}
// [{ "question_text": "...", "answers": ["...", "..."] }, ...]

## 8. Prompt 7 — Forge Works Map® Classification

**Job:** `fw_classify`
**Triggered:** After insight approval / enquiry close
**NOT applied to:** Individual observations — signal too thin
**Confidence threshold:** 0.70 — only factors meeting this threshold are stored
**Output:** Multi-factor parallel arrays on parent entity

### The 15 Factors

```
GUIDE (direction & context)
  senior_leadership      — How do senior leaders talk about and embody safety?
  strategy               — What triggers safety improvements?
  risk_management        — What is the quality of risk information and how is it used?
  safety_organisation    — How capable is the safety function?
  work_understanding     — What model of accident causation drives decisions?

ENABLE (resources & systems)
  operational_management    — What is the role of middle/frontline managers?
  resource_allocation       — How are safety resources identified and allocated?
  management_systems        — How effective are safety management systems?
  goal_conflict_tradeoffs   — How are safety goals balanced with production/cost?
  learning_development      — How does the organisation develop capability?

EXECUTE (frontline & operations)
  frontline_workers          — What is the role of frontline workers in safety outcomes?
  communications_coordination — How does information flow between teams?
  decision_making            — How are work and safety decisions made?
  contractor_management      — How are contractors engaged and managed?
  monitoring_metrics         — What information is used to monitor performance?
```

### Maturity Levels

```
compliant   — Rules, compliance, procedures prescribe work
leading     — Leadership behaviours, risk culture, safety climate
resilient   — Work-as-done, emergent risk, safety as property of work
```

### System Prompt

```
You are a safety management analyst trained in the Forge Works Map® — a 15-factor
organisational capacity framework.

Critical rules:
- Only classify a factor if the narrative provides direct evidence for that factor.
- Classify the ORGANISATIONAL factor — not the task or the hazard.
- Every factor must meet confidence >= 0.70 independently on its own evidence.
- If confidence is below 0.70, do not include that factor.
- Maximum 3 factors. If more than 3 meet the threshold, return the 3 with highest confidence.
- If no factor meets the threshold, return an empty classifications array.
- Each classification needs its own rationale — one sentence explaining why that specific
  factor was tagged based on specific evidence in the narrative.
- The threshold is the defence. A factor either meets 0.70 or it doesn't.

Output only valid JSON. No preamble, no markdown.
```

### User Prompt Template

```
Source type: {{source_type}}
// critical_insight | enquiry_summary

{{#if source_type == 'critical_insight'}}
Pattern summary: {{pattern_summary}}
Likely systemic cause: {{likely_systemic_cause}}
Toolbox narrative: {{toolbox_narrative}}
Work type: {{work_type_label}}
{{/if}}

{{#if source_type == 'enquiry_summary'}}
Enquiry title: {{enquiry_title}}
Synthesis findings: {{synthesis_findings_json}}
Summary narrative: {{summary_narrative}}
{{/if}}

Classify against the 15 Forge Works Map® factors.

GUIDE: senior_leadership, strategy, risk_management, safety_organisation, work_understanding
ENABLE: operational_management, resource_allocation, management_systems, goal_conflict_tradeoffs, learning_development
EXECUTE: frontline_workers, communications_coordination, decision_making, contractor_management, monitoring_metrics

Maturity: compliant | leading | resilient

Return JSON:
{
  "classifications": [
    {
      "fw_factor": "management_systems",
      "fw_domain": "guide | enable | execute",
      "fw_maturity_signal": "compliant | leading | resilient",
      "fw_confidence": 0.86,
      "fw_rationale": "1 sentence — why THIS factor based on THIS evidence"
    }
  ],
  "fw_classification_basis": "What specific evidence made this narrative classifiable at all",
  "attempted": true
}

// classifications is empty array if nothing met 0.70
// attempted = false if input was too thin to attempt (don't store, don't set fw_classified_at)
// Max 3 items, ordered by fw_confidence descending
```

### Validation Rules

- Every factor must have `fw_confidence >= 0.70` — reject any below threshold before storing
- `fw_factor` must be one of the 15 enumerated values
- `fw_domain` must be consistent with `fw_factor` — validate the mapping
- Store as parallel arrays: `fw_factors[i]`, `fw_domains[i]`, `fw_confidences[i]`, `fw_rationales[i]`
- Never display factor tags without their rationale — rationale is the defence
- `classifications = []` and `attempted = true` → store empty arrays, set `fw_classified_at` — do not retry
- `attempted = false` → store nothing, do not set `fw_classified_at` — re-queue when more context available

## 9. Error Handling Reference

```typescript
async function callAI(
  systemPrompt: string,
  userPrompt: string,
  maxTokens = 1000,
  imageBlocks?: ImageBlock[]
): Promise<unknown> {
  let rawText = ''
  try {
    const content = imageBlocks
      ? [...imageBlocks, { type: 'text' as const, text: userPrompt }]
      : userPrompt

    const response = await anthropic.messages.create({
      model: AI_MODEL_DEFAULT,
      max_tokens: maxTokens,
      system: systemPrompt,
      messages: [{ role: 'user', content }],
    })

    rawText = response.content
      .filter(b => b.type === 'text')
      .map(b => b.text)
      .join('')

    const clean = rawText.replace(/```json|```/g, '').trim()
    return JSON.parse(clean)

  } catch (error) {
    logger.error('AI prompt failed', { error, rawOutput: rawText })
    throw error  // Let BullMQ handle retry with backoff
  }
}
```

## 10. Prompt Versioning

```sql
CREATE TABLE safety_intelligence.ai_prompt_config (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  prompt_key  VARCHAR(50) NOT NULL,
  version     INTEGER NOT NULL,
  system_prompt TEXT NOT NULL,
  user_prompt_template TEXT NOT NULL,
  model       VARCHAR(50) NOT NULL DEFAULT 'claude-sonnet-4-6',
  max_tokens  INTEGER NOT NULL DEFAULT 1000,
  is_active   BOOLEAN NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ DEFAULT now(),
  UNIQUE (prompt_key, version)
);
```

Prompt keys: `observation.enrich`, `critical_insight.generate`, `toolbox_talk.generate`,
`enquiry.generate_questions`, `enquiry.synthesise`, `fw_classify`,
`situational_brief.generate` (V2), `visit_briefing.generate` (V2), `cop_thread.generate` (V3).

Active prompt selection: always the highest version where `is_active = true` for a given key.
