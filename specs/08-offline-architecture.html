Offline-First Capture
Architecture

How observation and incident capture works when connectivity is partial or absent. Observations are always written locally first. Classification runs async. The supervisor never waits.
V1 scope
Mobile-first (Capacitor)
April 2026
Core principles
01
Local write first
Every observation is written to the local database immediately on submission — before any network call is attempted. The supervisor's action is never blocked by connectivity.
02
Enrichment is async
AI classification (signal type, energy, Forge Works factor) runs in a background job after submission. The record exists and is valid without it. Enrichment updates the record when it completes.
03
Eventually consistent
Classification is eventually consistent — not real-time. A record moves through enrichment_status states (pending → processing → complete) on its own schedule. The user experience never degrades.

Enrichment retry schedule
T+0 — immediate
First enrichment attempt
Queued as BullMQ job immediately after sync. Runs within seconds on happy path.
T+5 min
Retry 2 — exponential backoff begins
enrichment_attempts incremented. Last error logged to enrichment_last_error.
T+15 min
Retry 3
Covers brief API outages and rate limit windows.
T+30 min · T+1h · T+2h · …
Retries 4–11 — extended backoff
Covers longer API degradation events. Observation is visible and usable throughout — just unclassified.
T+12 attempts
Final failure — escalate
enrichment_status → failed. Safety manager notified. Manual classification available in admin panel. Observation remains in pool — not lost.

Implementation considerations
Local storage
Use SQLite via Capacitor on device
@capacitor-community/sqlite gives a consistent SQL interface across iOS and Android. Mirror the server Prisma schema for the local tables that need offline support — observations, photos, sync_queue.
Sync queue
Simple FIFO with retry tracking
A local sync_queue table holds pending observations ordered by created_at. On reconnect, flush in order. Each item tracks attempt count and last error. Network reconnect events trigger flush via Capacitor Network plugin.
Conflict resolution
Last-write-wins on enrichment fields only
The observation text, work type, and stop_work_called are immutable after submission. Only enrichment fields are updated post-sync. No conflict risk — server is authoritative for enrichment, device is authoritative for submission content.
Enrichment worker
BullMQ with Redis — existing infrastructure
Enrichment jobs use the same BullMQ queue already specified in the V1 arch. Job payload: observation_id. Worker fetches fresh from DB, calls Anthropic API, writes results back. Idempotent — safe to retry.
Token cost
~$0.002 per enrichment call
Enrichment prompt is ~600 tokens on Sonnet ($3/M input, $15/M output). With photos: ~2,000–4,000 tokens depending on image count and size. At 1,000 observations/month: ~$2 without photos, ~$10–15 with photos.
Failed enrichment UX
Unclassified observations are still useful
A failed enrichment observation still appears in the safety manager's observation list — just without signal type or FW factor tags. It can be manually classified. It contributes to observation count. It is not silently dropped.
